# Anchor Day Model

An anchor day is a regular point in the week, fortnight, or month when people know digital research community activity is likely to happen.

It works well when several clubs or networks want visibility without forcing every group to run something every week.

## Purpose

An anchor day can:

- Make the community easier to find.
- Give people permission to protect time for learning and peer support.
- Create a rhythm for co-working, clinics, demos, and informal networking.
- Help new facilitators run small sessions with support.
- Reduce fragmentation across related clubs.

## Example Pattern

| Time | Activity | Notes |
| --- | --- | --- |
| 10:00-11:00 | Open co-working | Quiet shared work with helper support |
| 11:00-12:00 | Clinic | Bring technical questions |
| 12:00-13:00 | Lunch or informal networking | Optional and low pressure |
| 13:00-14:00 | Tutorial, demo, or reading group | Rotating topic |
| 14:00-15:00 | Leadership or planning hour | Convenors coordinate future sessions |

Do not start with a full-day programme unless there is enough capacity. A single recurring hour can still act as an anchor.

## Good Defaults

- Publish the term programme early.
- Mark beginner-friendly sessions clearly.
- Keep at least one low-preparation format in the rotation.
- Invite related clubs to take turns hosting.
- Use a shared calendar and shared notes.
- Review attendance and organiser workload every term.

## Risks

| Risk | Mitigation |
| --- | --- |
| Event fatigue | Vary formats and make attendance optional |
| Too much work for one organiser | Rotate hosts and keep reusable run sheets |
| Narrow audience | Use mapping data to invite underrepresented groups |
| Calendar clashes | Coordinate with related clubs before publishing |
| Sessions become too advanced | Protect regular beginner-friendly slots |
